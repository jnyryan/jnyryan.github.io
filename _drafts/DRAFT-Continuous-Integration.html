A very important aspect of modern software development is the concept of continuous integration (CI). It’s the practice of bringing all changes to a code base together as those changes are made - to ensure everything still behaves as expected. Done properly it brings the advantage of allowing developers to focus on feature development and away from the more mundane tasks like explaining to other developers how to set up your app on their machine, generating configuration files for specific environments or finding out your code breaks someone else’s feature a few hours before launch!

Take the typical example of a company that has multiple teams distributed globally working on e-commerce website. While one team works on the front-end UI, others will work on backend inventory modules and others on payment interfaces. These all have to work in unison and if not designed robustly have the unfortunate possibility of affecting each others functionality and/or performance with the worst result possible - lost revenue and a very “shouty” call from the CTO. CI aims to reduce these undesirable outcomes by automating the common points of failure. The underlying premise behind CI is “If it’s repeatable – script it!”

It brings together three other concepts – Continuous Build, Continuous Test, Automatic Deployment

Automatic Deployment

 

CI solution should answer the following

  • Can I get the latest code from source control and get developing with minimal fuss
  • Can I run the entire app locally and not require external help to get bug fixing
  • Can a member of the QA team deploy this to an environment with one-click or one command line
  • If I make a change to code can I find out if I did any harm quickly
  • If I create a new feature can I deploy it to production with high confidence that it will not impact the rest of the application.
  • Am I necessary to deploy the app.
